# Authoring Intake

Mission / stance:
Turn a named skill idea into a usable authoring receipt without starting a
portfolio audit.

When to use:
- The target skill/change is named but underspecified.
- The user gives examples without a reusable job, owner plugin, or proof shape.
- The prompt mixes one concrete skill with broad inventory or merge curiosity.

What to inspect:
- proposed skill/change
- user examples and acceptance source
- nearby skill surface and owner plugin
- baseline failure hypothesis or proof gap
- first plausible proof shape

How to inspect:
Classify the run as `create`, `update`, or `evaluate`; name the reusable job;
identify the owner plugin and expected target files; check whether a current
skill already owns the job; state the baseline failure or proof-gap reason; and
choose the first proof shape.

Good signals:
- one target skill/change
- clear owner plugin
- reusable job stated in one sentence
- existing-surface check done before creating anything new
- broad inventory deferred or routed separately

Bad signals:
- treating overlap curiosity as permission for broad `skill-audit`
- creating a new skill before checking current ownership
- no baseline failure or proof gap
- proof shape hidden until after editing
